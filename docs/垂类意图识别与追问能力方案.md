# 游戏中心「垂类意图识别 / 追问能力」方案

> 模块定位：搭建服务于游戏中心 C 端功能和业务的用户意图识别体系，做好连接用户的第一站。设计追问策略与话术，引导用户补全模糊需求，支持多轮对话。
>
> 撰写日期：2026-03-25 | 更新日期：2026-03-26

---

## 一、整体架构

四层流水线：**预处理 -> 意图识别 -> 追问决策 -> 响应生成**

- **Layer 0 预处理**：谐音纠错、缩写展开、标点去噪、多模态转文本
- **Layer 1 意图识别**（双引擎）：规则引擎（即时，强关键词/正则直接命中）+ LLM 语义理解（模糊输入大模型分类）+ 上下文融合（对话历史+用户画像）
- 置信度打分：高(>=85%)直接执行 / 中(60-84%)确认 / 低(<60%)追问
- **Layer 2 追问决策**：Slot 检查 -> 策略选择 -> 话术匹配
- **Layer 3 响应生成**：结构化卡片 + quickReplies + 兜底拉回

## 二、意图清单：如何确定要识别什么

### 2.1 定义原则

意图不是拍脑袋想的，要通过**穷举法 + 聚类**：

1. **收集真实语料**：找 20 人自由提问，收集 200-300 条真实提问
2. **人工打标聚类**：逐条标注用户到底想要什么，自然聚堆
3. **对齐服务能力**：每个意图必须对应一个明确的下游服务，没有服务能力的不进清单

### 2.2 C 端意图体系（三级架构）

**一级意图（5大类）**：信息获取、功能执行、社交互动、情感表达、闲聊/域外

**二级意图（14个）**：

| 意图 ID | 标签 | 归属 | 识别出来后干什么 |
|---------|------|------|---------------|
| record | 战绩查询 | 信息获取 | 调战绩 API 返回数据卡片 |
| match_query | 对局查询 | 信息获取 | 拉对局列表展示 |
| replay | AI 复盘 | 信息获取 | 调复盘服务分析单局 |
| news | 游戏资讯 | 信息获取 | 拉资讯流/LLM 摘要 |
| guide | 攻略查询 | 信息获取 | 调攻略库/LLM 生成 |
| ranking | 好友排行 | 信息获取 | 拉排行数据展示 |
| report | 日报/周报 | 信息获取 | 聚合周期数据 |
| welfare | 福利/礼包 | 功能执行 | 拉活动列表展示 |
| download | 游戏下载 | 功能执行 | 跳转下载页 |
| skin | 商城 | 功能执行 | 跳转商城/推荐皮肤 |
| reminder | 定时提醒 | 功能执行 | 创建提醒任务 |
| highlight | 高光视频 | 功能执行 | 调视频生成服务 |
| partner | 找搭子 | 社交互动 | 调匹配系统推荐队友 |
| emotion | 情绪互动 | 情感表达 | LLM 生成共情回复 |

**三级意图（子场景举例）**：以 record 为例 -> 总览查询 / 胜率查询 / 段位查询 / 英雄战绩 / 对比查询

## 三、意图识别技术方案

### 3.1 分层识别架构

不是一把梭，而是分层处理：
- **规则引擎处理 60-70% 的请求**：成本低、速度快、可控
- **LLM 处理剩余 30-40%**：语义理解强、处理模糊输入
- **上下文融合层**：减少 50% 以上的不必要追问

### 3.2 规则引擎（即时响应）

| 匹配层 | 策略 | 权重 | 示例 |
|--------|------|------|------|
| 强关键词 | 精确包含直接命中 | +5 | "战绩" -> record |
| 弱关键词 | 模糊包含累加 | +2 | "赢了多少" -> record |
| 正则模式 | 句式匹配 | +4 | 看看...战绩 -> record |
| 编辑距离 | 容错1-2字 | +2 | "战机" -> "战绩" |
| 缩写映射 | 拼音展开 | +8 | "zj" -> "战绩" |

**得分决策**：>=5 直接判定 / 3-4 判定但标记需确认 / <3 进 LLM

### 3.3 LLM 语义理解（1-3秒）

**触发条件**：规则引擎得分 < 3，或 Top1 和 Top2 得分差 <= 2（歧义）

**Prompt 核心要点**：
1. 严格 JSON 输出：intentId, confidence, gameId, missingSlots, needFollowUp
2. 理解口语/网络用语/谐音错别字
3. Few-shot 覆盖典型 case（清晰/模糊/歧义/情绪）
4. 不编造数据，不确定时给低 confidence + top2 候选

### 3.4 上下文融合（关键）

维护最近5轮对话的游戏/英雄标记，优先查标记再从文本重新识别。能从上下文补全的信息不再追问。

### 3.5 关键识别能力

| 能力 | 说明 |
|------|------|
| 游戏识别 | 全名/简称/黑话/拼音/上下文（农药->王者，吃鸡->和平） |
| 英雄识别 | 正名/别名/谐音/反向推断游戏（李白->王者荣耀） |
| 时间解析 | 今天/昨天/上一把/最近(7日)/这周/这个月 |
| 功能x游戏感知 | AI复盘仅王者 / 战绩支持王者+和平+三角洲 / 不支持时引导替代 |

## 四、Slot 体系与追问决策

### 4.1 各意图 Slot 表

| 意图 | 必填 Slot | 缺了怎么办 | 默认值 |
|------|----------|-----------|-------|
| record 战绩 | gameId | 追问或用最近活跃游戏 | 时间默认近7日 |
| replay 复盘 | gameId | 仅支持王者可直接默认 | 默认最近一局 |
| guide 攻略 | 无必填 | 直接展示热门攻略 | - |
| partner 找搭子 | gameId | 必须追问 | - |
| welfare 福利 | 无必填 | 展示全部可领福利 | - |
| emotion 情绪 | 无 | 直接回应不需要参数 | - |
| reminder 提醒 | time | 追问时间 | - |
| download 下载 | gameId | 追问游戏名 | - |

### 4.2 追问决策树

**核心原则：能不问就不问，必须问时只问一次。**

判断链：
1. 上下文有？静默补全（0追问）
2. 有默认值？用默认值执行+附修正选项（0.5追问）
3. 置信度>=70%？猜测执行+确认（0.5追问）
4. 都不行？选择式追问，带quickReplies（1追问）

**硬性规则**：最多2轮、每次必带选项、同一信息不问两次、情绪场景永远不追问

### 4.3 四种追问策略

| 策略 | 条件 | 示例 |
|------|------|------|
| A 静默补全 | 上下文/默认值可填 | 之前聊王者->帮我复盘->直接复盘王者 |
| B 猜测+行动 | 置信度>=70% | 看看我的数据->直接出王者战绩+想看其他也可以说 |
| C 选择式追问 | 不确定但选项有限 | 帮帮我->想查战绩找搭子还是看攻略？ |
| D 情绪优先 | 检测到情绪 | 气死了连跪->先共情再用quickReplies引导 |

## 五、话术设计规范

### 5.1 话术风格铁律

| 铁律 | 做法 | 反例 |
|------|------|------|
| 口语化 | 想看哪个游戏的？ | 请选择您要查询的游戏 |
| 永远带选项 | quickReplies 2-3个 | 开放式你想干什么？ |
| 先接住再问 | 连跪辛苦了，复盘看看？ | 检测到情绪波动是否需要... |
| 收窄不发散 | 给具体选项 | 开放式追问 |
| 先做后确认 | 用默认值出结果+不对再说 | 先把参数全问齐 |

### 5.2 多版本话术模板

| 场景 | 通用版 | 缩窄版 | 上下文版 |
|------|-------|--------|--------|
| 缺游戏名 | 想看哪个游戏的？ | 王者还是和平？ | 查的是王者荣耀吧？ |
| 缺时间 | 看最近一周的还是某一局？ | 今天还是最近几天？ | 看今天的对吧？ |
| 意图模糊 | 查战绩看复盘还是找攻略？ | 看数据还是吐个槽？ | - |

### 5.3 域外话题拉回

| 触发场景 | 拉回话术 | 引导功能 |
|---------|---------|--------|
| 累/困/烦 | 听起来需要放松，找搭子开黑？ | partner |
| 无聊/摸鱼 | 无聊？一把王者走起 | partner |
| 吃饭/奶茶 | 吃饭前先领个礼包 | welfare |
| 上班/加班 | 摸鱼看看本周战绩报告 | report |
| 666/nb | 把精彩操作做成高光视频 | highlight |

原则：先接住再拉回、找类比联系、永远给功能落点、幽默不冒犯

## 六、多轮对话与上下文管理

**三层记忆**：

| 层级 | 内容 | 用途 |
|------|------|------|
| 对话历史（近5轮） | 消息+gameId/intentId+追问状态 | 上下文补全、话题切换检测 |
| 会话级 | 当前活跃游戏/意图+已填Slot | Slot自动补全 |
| 用户级（长期） | 最近在玩+常用英雄+偏好 | 默认值推断 |

**话题切换**：不同意图强关键词->切换 / 算了->放弃 / 只说游戏名->继续补全

**指代消解**：省略游戏->取lastMentionedGame / 那个->取最近游戏 / 也->保留意图切换游戏

## 七、如何判断引导是否正确

### 7.1 看用户行为（最直接的信号）

正确引导信号：
- 用户点了quickReplies（选项设计合理）
- 用户打字补充了缺失信息（追问问对了）
- 用户完成了后续操作（端到端成功）

错误引导信号：
- 用户直接走了（追问让人烦了）
- 用户说不是或算了（猜错了）
- 用户重复表达同一意思（没理解用户）
- 用户换了完全不同话题（用户放弃了）

### 7.2 核心指标

| 指标 | 好 | 差（需优化） |
|------|---|------------|
| 意图识别准确率 | >=90% | <80% |
| 追问响应率 | >=70% | <50% |
| quickReplies点击率 | >=40% | <20% |
| 追问后完成率 | >=80% | <60% |
| 追问放弃率 | <=15% | >30% |
| 平均追问轮次 | <=1.3 | >2 |

### 7.3 必须埋的4个点位

| 埋点 | 记录什么 | 价值 |
|------|---------|------|
| 追问触发 | 什么意图、缺什么Slot、什么策略 | 知道追问发生在哪 |
| 追问回应 | 有没有回应、方式 | 知道追问有没有用 |
| 任务完成 | 追问后有没有完成需求 | 端到端成功率 |
| 隐式满意度 | 对话轮次、是否主动结束 | 整体体验评估 |

### 7.4 每周人工审核

每周抽50条追问对话，评审5个问题：
1. 这次追问有必要吗？能不能从上下文补全？
2. 追问的问题对不对？
3. 话术自然吗？
4. 选项有没有覆盖用户真正想要的？
5. 最终有没有解决问题？

## 八、追问策略与话术设计方法论

> 本章提供从0到1设计追问策略和话术的完整步骤，可作为团队协作落地的执行指南。

### Step 1：穷举用户会怎么说 - 收集真实语料

**目的**：知道用户到底怎么表达，才能知道哪里会听不懂。

**做法**：
- 针对每个意图，团队每人扮演用户写20句话（越口语越好，越模糊越好）
- 分三类收集：清晰表达（信息完整不需追问）、模糊表达（缺游戏缺时间）、极简/歧义（什么都缺）
- 如有对话日志直接提取真实bad case

**产出**：每个意图的语料表，标注缺失的Slot。

### Step 2：定义每个意图的 Slot 表

**目的**：明确每个意图的必要信息清单。

| 字段 | 是否必填 | 缺了会怎样 | 有没有默认值 |
|------|---------|-----------|------------|
| gameId | 必填 | 不知道查哪个游戏 | 用最近活跃游戏 |
| timeRange | 可选 | 默认近7天 | 近7日 |
| heroId | 可选 | 展示全部英雄 | 无 |

关键判断：哪些Slot一定要问、哪些可以猜、哪些不用管。

### Step 3：设计追问决策树

**目的**：把追问变成一套可编程的规则。

对每个缺失Slot走判断链：
1. 上下文有？自动补全
2. 有默认值？先用默认值出结果
3. 置信度够高？猜+确认
4. 都不行？追问带选项

关键规则：同时缺多个Slot只追问最关键的，最多2轮，情绪场景不追问。

### Step 4：编写话术模板

**目的**：让追问自然、不烦人。

每个追问场景写3个版本（通用/缩窄范围/有上下文暗示）。

**话术5条铁律**：口语化、永远带选项、先接住再问、收窄而非发散、多版本匹配。

### Step 5：模拟对话测试

角色扮演测试，重点测边界case：连续缺多个Slot、用户不理追问、答非所问、情绪场景、上下文切换。

**产出**：bad case清单和话术修改建议。

### Step 6：量化评估+持续迭代

核心指标：追问后解决率>=85%、平均追问轮次<=1.5、quickReplies点击率>=40%、追问放弃率<=20%。

每周看埋点数据，找放弃率最高的场景优先优化，A/B测试不同话术版本。

### 总结：6步流程

Step 1 收集语料 -> Step 2 定义Slot表 -> Step 3 设计决策树 -> Step 4 编写话术模板 -> Step 5 模拟对话测试 -> Step 6 上线埋点+迭代

> 讨论建议：团队讨论时从Step 1开始，让每个人现场扮演用户对着14个意图写模糊表达。

## 十、讨论议题

### 必须达成共识

1. **意图粒度**：14个二级意图够吗？需要新增活动查询、游戏推荐吗？
2. **引擎选型**：规则+LLM混合方案？LLM用DeepSeek还是混元？
3. **追问上限**：最多几轮？建议2轮
4. **默认值策略**：缺游戏名时追问 vs 用最近活跃游戏？
5. **验证标准**：追问放弃率多少算合格？

### 延伸讨论

- 截图识别匹配攻略
- 群场景 vs 私聊追问差异
- 冷启动用户的默认策略
